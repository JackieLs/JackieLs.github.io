CommonJS、AMD、CMD、UMD、ES6 Modules 的种种不同

2018-01-30 • 分类: 大前端 • 标签: javascript, nodejs, 前端模块化

前端模块化现在已经不是什么新的话题了，但是自己之前对这部分一直不是十分清晰，最近在看 BAT 相关面试的题目，其中就有关于这几种规范的一些题目，今天对这部分内容进行一个总结。

目前的规范主要有以下几种

  • CommonJS
  • AMD (Asynchronous Module Definition)
  • CMD (Common Module Definition)
  • UMD (Universal Module Definition)
  • ES6 Modules

每一种规范的出现都代表了一个时代，一种思维，都有对应的具体实践，下面我们来一点点剖析。

CommonJS

CommonJS 根据维基百科的说法，是在 2009 年一月诞生，那时候叫 ServerJS，后来在八月份正式更名为 CommonJS，这个规范的出现更多的是为了解决团队协作的问题。

What I’m describing here is not a technical problem. It’s a matter of people getting together and making a decision to step forward and start building up something bigger and cooler together — Kevin Dangoor

2013 年的时候，npm 的作者说这个规范正在被 NodeJS 慢慢淘汰。(WTF? 都 2018 了，再过几年升级 Node 就要改代码咯？)原因很简单，因为 CommonJS 是野路子，并没有被 ECMA 国际组织 TC39 纳入规范，而真正的官方规范是 ES6 Modules 后面会细说。

AMD 规范

翻译一下就知道 AMD 是异步模块加载，目前流传比较广泛的使用该规范的是 RequireJS，该规范的意义就是在浏览器端实现了模块的异步加载，也就是<script async></script>。

提前加载所有的模块，AMD 运行时核心思想是「Early Executing」，也就是提前执行依赖，把所有的依赖加载完之后，再去执行之后的 callback，好处：

  • 保证运行不会出错，不会出现运行到某一个代码的时候依赖没有加载出来
  • 不会出现页面的假死，html 和 css 加载出来之后，不会因为某些 js 没有加载出来导致整个页面卡在一个空白的地方，而是继续执行后面的内容。

缺点的话，之前浅浅的接触过一个用 Backbone + RequireJS 写的项目，觉得缺点主要有两点：

  • 写起来很费劲
  • 无法实现按需加载

CMD 规范

因为无法按需加载，所以阿里的玉伯就弹出来这个 CMD 规范，搞了个 SeaJS，这个我是一点也没用过，2015 年年末的时候接触了一下 RequireJS 和 Backbone 就直接去搞 React 了，见证了 React 的发展强大。不过这个 Seajs 正经也是很多人的青春啊。
跟 AMD 最不一样的地方就是可以按需加载，哪里想用就在哪里直接 require 就行了

好处是按需加载，不好的地方就是一旦包多了，比如 50 个，那么就会发 50 个请求，这么多请求所浪费的网络资源，还不如不按需加载，那怎么办，就得打包了，SPM 啊，grunt 啊，都是上个时代的东西了，我是只听过没见过啊。有兴趣的请看鑫富帅：高富帅 seajs 使用示例及 spm 合并压缩工具露脸
总结起来就是：
AMD：提前执行，依赖前置，全部加载。
CMD：延迟执行，依赖就近，按需加载。

写在结尾

总结一下

  • CommonJS 同步运行，可动态加载，模块缓存，主要运行在 Server 端
  • AMD Client 端的 CommonJS，异步加载 JS，提前执行，依赖前置
  • CMD 优化了 AMD，异步加载 JS，延迟执行，依赖就近，按需加载
  • UMD 兼容 CommonJS 和 AMD 的兼容(CommonJS:typeof exports == "object" && typeof module == "object";AMD:typeof define === 'function' && define.amd)
  • ES6 Module 官方标准，解构引用是对目标模块对象的引用，整体引用就是对目标模块对象的 default 属性的引用，依赖提升，只能有一个export default，不支持动态加载。

通过对比了 CommonJS、AMD、CMD、UMD 和 ES6 Modules，我们不难发现，所有的规范都是通过 require 函数和 js 的特性 (回调，闭包，对象指针等等) 变形来的，但是因为种种特性导致了代码运行的过程完全不一样。
